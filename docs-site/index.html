The Problem

Agents have unlimited power

AI agents can send emails, run shell commands, transfer money, and deploy to production ‚Äî but there's no universal way to ensure a human said "yes" first.

Every agent framework has its own half-baked "human-in-the-loop" that the agent can bypass. Give an agent shell access, and it has the same power as root.

üíÄ

Shell Access = Root

An agent with exec() can run anything. Install packages, delete files, exfiltrate data. No guardrails.

üìß

Credential Theft

API keys in environment variables? The agent can read them. And send them anywhere.

üí∏

Financial Risk

Stripe keys, bank APIs, payment webhooks. One bad tool call could cost real money.

üîì

Bypassable Controls

Framework-level "approval" lives in the same process as the agent. It can be skipped programmatically.

How It Works

Network-level agent containment

ACP runs your agent in a Docker container with no direct internet. Shell commands and HTTP requests are intercepted by two layers. The agent can't bypass it ‚Äî the container has no way out except through ACP.

Policy Engine

Fine-grained control rules

Define exactly what your agent can and can't do. Allow reads, gate shell access, block financial actions ‚Äî all in a simple YAML config.

yaml ‚Äî acp-policy.yml 
default: ask

wrap:
  - gh
  - curl
  - rm

rules:
  - match: { name: "cat" }
    action: allow

  - match: { kind: http, host: "*.anthropic.com" }
    action: allow

  - match: { name: "rm", args: "-rf /*" }
    action: deny

How policies work

ACP evaluates rules top-to-bottom. The first matching rule wins. Unmatched actions fall through to the default.

  • ‚úì allow ‚Äî Action executes immediately, no prompt
  • ‚ö† ask ‚Äî You get a notification and must approve
  • ‚úï deny ‚Äî Blocked unconditionally. Agent gets an error.

The wrap list controls which commands get shell wrappers (Layer 1). All HTTP traffic goes through the proxy (Layer 2).
